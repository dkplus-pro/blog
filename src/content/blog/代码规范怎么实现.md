要确保一个前端项目同时兼容 **Windows / Mac / AI 写代码**，不能只靠“提交阶段自动格式化”。最佳做法是做成 4 层：

```text
1. 编辑器层：保存时统一格式
2. 工具层：Prettier / ESLint / Stylelint 强制规范
3. Git 层：提交前校验、换行符统一、commit 规范
4. AI 层：Cursor / Claude / Codex 明确项目规则
```

## 一、必须要有的工具

| 工具                | 作用                     |
| ----------------- | ---------------------- |
| EditorConfig      | 统一不同编辑器的缩进、换行、编码       |
| Prettier          | 统一格式：引号、分号、换行、尾逗号      |
| ESLint            | 检查 JS/TS/Vue 代码问题      |
| TypeScript        | 类型检查                   |
| Stylelint         | CSS/SCSS/样式规范，可选但推荐    |
| Husky             | Git hooks，提交前自动执行命令    |
| lint-staged       | 只检查本次提交的文件             |
| Commitlint        | 规范 commit message      |
| `.gitattributes`  | 统一 Windows/Mac 的 LF 换行 |
| AI Rules / Skills | 约束 AI 按项目规则写代码         |

EditorConfig 本来就是为跨编辑器保持一致编码风格设计的；Prettier 也会读取 `.editorconfig` 里的 `end_of_line`，除非被 Prettier 配置覆盖；ESLint 负责发现 JS/TS 代码中的问题并让代码更一致；lint-staged 的核心就是只对 Git 暂存区文件运行格式化或 lint 任务；Husky 则负责在 commit/push 阶段自动跑检查。([editorconfig.org][1])

## 二、项目目录里应该有什么

建议前端项目根目录至少有这些：

```text
project/
  .editorconfig
  .gitattributes
  .gitignore
  .prettierrc
  .prettierignore
  eslint.config.js
  stylelint.config.js
  commitlint.config.js
  package.json
  tsconfig.json

  .vscode/
    settings.json
    extensions.json

  .cursor/
    rules/
      frontend.mdc
      ai-coding.mdc

  .claude/
    skills/
      frontend-project/
        SKILL.md

  .husky/
    pre-commit
    commit-msg

  src/
```

如果你们只用 Cursor，可以先放 `.cursor/rules`；如果你们也用 Claude Code，就放 `.claude/skills`。Cursor 官方支持用 Project / Team / User Rules 给 AI 配持久指令；Claude Skills 也是用来打包领域知识和组织工作流的自定义能力。([Cursor][2])

## 三、跨 Windows / Mac 最关键：统一 LF

### 1. `.editorconfig`

```ini
root = true

[*]
charset = utf-8
end_of_line = lf
insert_final_newline = true
indent_style = space
indent_size = 2
trim_trailing_whitespace = true

[*.md]
trim_trailing_whitespace = false
```

作用：

```text
让不同编辑器保存文件时默认使用 LF、UTF-8、2 空格缩进
```

---

### 2. `.gitattributes`

```text
* text=auto eol=lf

*.png binary
*.jpg binary
*.jpeg binary
*.gif binary
*.webp binary
*.ico binary
*.pdf binary
*.zip binary
```

作用：

```text
Git 层面强制文本文件用 LF
避免 Windows 下反复出现 CRLF / LF 变化
```

Windows 开发者还建议执行：

```bash
git config --global core.autocrlf input
```

不要用：

```bash
git config --global core.autocrlf true
```

否则可能出现：

```text
LF will be replaced by CRLF
```

---

### 3. `.vscode/settings.json`

Cursor 也兼容 VSCode 设置：

```json
{
  "files.eol": "\n",
  "editor.formatOnSave": true,
  "editor.defaultFormatter": "esbenp.prettier-vscode",
  "eslint.validate": ["javascript", "typescript", "vue"],
  "editor.codeActionsOnSave": {
    "source.fixAll.eslint": "explicit"
  }
}
```

作用：

```text
新建文件默认 LF
保存时自动格式化
保存时可手动触发 ESLint fix
```

## 四、Prettier 配置

`.prettierrc`：

```json
{
  "printWidth": 120,
  "tabWidth": 2,
  "useTabs": false,
  "semi": false,
  "singleQuote": true,
  "quoteProps": "as-needed",
  "bracketSpacing": true,
  "trailingComma": "none",
  "bracketSameLine": false,
  "arrowParens": "always",
  "vueIndentScriptAndStyle": true,
  "htmlWhitespaceSensitivity": "ignore",
  "endOfLine": "lf"
}
```

`.prettierignore`：

```text
node_modules
dist
build
coverage
unpackage
.husky
*.min.js
*.lock
```

重点是：

```json
"endOfLine": "lf"
```

这个专门解决 Windows / Mac 换行不一致。

## 五、ESLint 配置

如果是 Vue / uniapp / TS 项目，常见依赖：

```bash
pnpm add -D eslint prettier eslint-config-prettier eslint-plugin-vue typescript typescript-eslint
```

如果你们还是旧 `.eslintrc.cjs` 也可以，但新项目更建议 `eslint.config.js`。TypeScript ESLint 官方也已经提供 flat config 的快速开始方式。([typescript-eslint.io][3])

基础思路：

```text
ESLint 管代码质量
Prettier 管代码格式
eslint-config-prettier 关闭和 Prettier 冲突的规则
```

不建议让 ESLint 和 Prettier 都管缩进、换行、引号，不然会打架。

## 六、Stylelint 配置

如果项目 CSS / SCSS / Vue style 较多，建议加：

```bash
pnpm add -D stylelint stylelint-config-standard stylelint-config-recess-order
```

`stylelint.config.js`：

```js
export default {
  extends: ['stylelint-config-standard', 'stylelint-config-recess-order'],
  ignoreFiles: ['**/node_modules/**', '**/dist/**', '**/build/**', '**/unpackage/**'],
  rules: {
    'selector-class-pattern': null,
    'no-descending-specificity': null
  }
}
```

如果你们用 UnoCSS 很多，Stylelint 可以先轻一点，不要管太死。

## 七、package.json 脚本

建议至少有这些：

```json
{
  "scripts": {
    "dev": "vite",
    "build": "vite build",
    "typecheck": "vue-tsc --noEmit",
    "lint": "eslint . --quiet",
    "lint:fix": "eslint . --fix --quiet",
    "format": "prettier . --write",
    "format:check": "prettier . --check",
    "stylelint": "stylelint \"src/**/*.{css,scss,vue}\"",
    "stylelint:fix": "stylelint \"src/**/*.{css,scss,vue}\" --fix",
    "check": "pnpm typecheck && pnpm lint && pnpm format:check"
  }
}
```

日常开发：

```bash
pnpm lint:fix
pnpm format
```

CI / 合并前：

```bash
pnpm check
```

## 八、提交前自动规范

安装：

```bash
pnpm add -D husky lint-staged @commitlint/cli @commitlint/config-conventional
pnpm exec husky init
```

`package.json`：

```json
{
  "lint-staged": {
    "*.{js,jsx,ts,tsx,vue}": [
      "eslint --fix --quiet",
      "prettier --write"
    ],
    "*.{css,scss,less,vue}": [
      "stylelint --fix",
      "prettier --write"
    ],
    "*.{json,md,yml,yaml}": [
      "prettier --write"
    ]
  }
}
```

`.husky/pre-commit`：

```bash
pnpm lint-staged
```

`commitlint.config.js`：

```js
export default {
  extends: ['@commitlint/config-conventional']
}
```

`.husky/commit-msg`：

```bash
pnpm commitlint --edit "$1"
```

这样 AI 写出来的代码，即使有格式问题，也会在提交前被兜底处理。

## 九、AI 写代码要不要特别声明？

要。不能只靠提交阶段。

因为如果只靠提交阶段，AI 可能会：

```text
改太多无关文件
顺手重构
格式化整仓库
引入 any / Function / 未使用变量
把目录放错
不按项目封装方式写
```

所以要同时做：

```text
AI 规则提前约束
工具链提交前兜底
CI 最后强制检查
```

## 十、Cursor 规则建议

`.cursor/rules/frontend.mdc`：

```md
---
description: Frontend project coding rules
globs:
  - "src/**/*"
alwaysApply: true
---

# Frontend Coding Rules

## General

- Follow the current project's ESLint, Prettier, TypeScript and Stylelint configuration.
- Make minimal changes.
- Do not refactor unrelated code.
- Do not format unrelated files.
- Do not change business logic unless explicitly requested.
- Do not introduce new lint errors.
- Do not use `any` unless necessary and explain why.
- Do not use `Function` as a type.
- Remove unused imports and unused variables.

## File Placement

- `api/` only contains request functions.
- `services/` contains business orchestration.
- `store/` contains global shared state.
- `types/` contains TypeScript types only.
- `enums/` contains runtime enum values.
- `consts/` contains stable constants.
- `config/` contains environment and module configuration.
- `utils/` contains pure utility functions.
- `components/` contains UI components.

## Before Finishing

- Run lint or typecheck for changed files when possible.
- If a command fails, explain the reason and fix only task-related issues.
```

## 十一、Claude Code Skill 建议

`.claude/skills/frontend-project/SKILL.md`：

```md
---
name: frontend-project
description: Use this skill when modifying frontend code, adding modules, fixing lint errors, creating components, or changing project structure.
---

# Frontend Project Rules

## Code Quality

- Follow ESLint, Prettier, TypeScript and Stylelint configs.
- Prefer minimal changes.
- Do not refactor unrelated code.
- Do not format unrelated files.
- Do not introduce lint errors.
- Do not use `any` unless unavoidable.
- Do not use `Function` as a type.
- Remove unused imports and variables.

## Architecture

- api only contains request functions.
- services contain business workflows.
- store contains global shared state.
- types contain TypeScript types.
- enums contain runtime enum values.
- consts contain constants.
- config contains project or environment configuration.
- utils contain pure helpers.
- composables/hooks contain reusable logic.
- components contain UI components.

## Verification

After changes, prefer:
- eslint on changed files
- prettier on changed files
- typecheck when types are changed
```

## 最终结论

你的目标应该是：

```text
EditorConfig + Prettier + ESLint + Stylelint
解决代码风格

.gitattributes + files.eol + core.autocrlf=input
解决 Windows / Mac 换行

Husky + lint-staged + Commitlint
解决提交质量

Cursor Rules + Claude Skills
解决 AI 写代码不按规范

CI check
解决最终强制兜底
```

一句话：**AI 要声明规则，工具要自动修，Git 要统一换行，CI 要最终拦截。**
