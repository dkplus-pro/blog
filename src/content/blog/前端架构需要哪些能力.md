# 前端能力体系建设文档

## 1. 文档目的

本文用于梳理前端团队在工程化、稳定性、性能优化、质量保障、安全、架构设计、研发协作、业务通用能力以及 AI 辅助研发等方面的能力建设方向。

目标不是一次性全部完成，而是建立一份可持续演进的前端能力地图，用于指导团队后续的规范建设、平台接入、自动化治理和研发效率提升。

---

## 2. 总体能力地图

前端能力体系可以分为以下九个方向：

```txt
1. 工程化
2. 稳定性
3. 性能优化
4. 质量保障
5. 安全
6. 架构设计
7. 研发协作
8. 业务通用能力
9. AI 辅助研发
```

其中：

* **工程化** 解决“如何高效、稳定地开发和发布”；
* **稳定性** 解决“线上出问题后如何发现、定位和治理”；
* **性能优化** 解决“如何让页面更快、更顺、更省资源”；
* **质量保障** 解决“如何在上线前减少问题”；
* **安全** 解决“如何避免敏感信息、权限、依赖和运行时风险”；
* **架构设计** 解决“复杂业务如何拆分、组织和长期维护”；
* **研发协作** 解决“多人协作、评审、上线和复盘流程”；
* **业务通用能力** 解决“重复业务能力如何沉淀复用”；
* **AI 辅助研发** 解决“如何让 AI 稳定参与编码、Review、Debug 和文档建设”。

# 3. 工程化

工程化是前端能力体系的基础，主要解决项目规范、构建发布、平台协同和自动化流程问题。

## 3.1 项目本身

项目本身关注代码如何组织、如何约束、如何构建，以及如何让团队成员用一致的方式开发。

### 3.1.1 代码规范

包括：

* TypeScript 规范；
* Vue / uni-app 代码规范；
* 组件命名规范；
* 变量命名规范；
* 函数拆分规范；
* 注释规范；
* 单文件行数规范；
* 单函数行数规范；
* Store 拆分规范；
* Hooks / Composables 使用规范；
* Biz / Controller 业务编排层规范。

建议约束：

```txt
普通业务文件建议控制在 300 行以内；
Vue 页面文件建议控制在 500 行以内；
Store 文件超过 500 行必须评估拆分；
单个函数建议控制在 80 行以内；
超过 800 行的文件原则上必须拆分；
超过 1000 行的文件视为高风险文件。
```

### 3.1.2 提交规范

包括：

* commit message 规范；
* commitlint；
* husky；
* lint-staged；
* merge commit 规范；
* release commit 规范；
* tag 命名规范。

推荐提交格式：

```txt
feat: 新增功能
fix: 修复问题
docs: 文档变更
style: 样式调整
refactor: 代码重构
test: 测试相关
chore: 工程配置
build: 构建相关
ci: CI/CD 相关
```

### 3.1.3 Gitflow

适合轻量团队的分支模型：

```txt
main
  生产稳定分支，只保留已上线代码

release
  提测与预上线分支，push 后自动触发测试构建

feature/xxx
  日常需求开发分支

bugfix/xxx
  普通 bug 修复分支

hotfix/xxx
  生产紧急修复分支

tag
  生产上线成功后的版本标记
```

推荐规则：

* 正式 tag 只打生产版本；
* 测试环境默认不打正式 tag；
* 生产上线成功后，再在最终生产 commit 上打 tag；
* 已推送 tag 禁止随意删除或覆盖。

### 3.1.4 IDL 生成代码

IDL 用于接口协议驱动开发，减少手写类型和接口请求代码。

可建设内容：

* IDL 自动生成 TypeScript 类型；
* IDL 自动生成 API 请求函数；
* IDL 自动生成 Mock 数据；
* IDL 变更检查；
* 接口破坏性变更提示；
* 前后端接口契约校验。

### 3.1.5 项目规范

包括：

* 目录结构规范；
* 模块分层规范；
* 页面开发规范；
* 组件开发规范；
* API / Service 规范；
* Store 规范；
* Hooks / Composables 规范；

# 4. 稳定性

稳定性关注线上问题的发现、定位、通知、修复和复盘闭环。

## 4.1 错误采集

需要采集的错误类型：

* JS Error；
* Promise 未捕获异常；
* Vue errorHandler；
* 接口异常；
* 资源加载失败；
* 白屏异常；
* 路由异常；
* 小程序生命周期异常；
* 小程序 setData 异常；
* 用户主动反馈。

## 4.2 错误上下文

错误上报时应尽量携带：

* 用户 ID；
* 设备信息；
* 系统版本；
* 应用版本；
* 构建版本；
* 页面路径；
* 路由参数；
* 接口 traceId / logId；
* 最近操作路径；
* 请求参数脱敏；
* 接口返回脱敏；
* 错误堆栈；
* Sourcemap 还原信息。

## 4.3 错误看板

看板需要支持：

* 错误量；
* 错误率；
* 影响用户数；
* 影响页面；
* 影响版本；
* 影响环境；
* Top 错误；
* 新增错误；
* 复现路径；
* 错误趋势；
* 接口失败率；
* 接口耗时；
* 白屏率；

## 4.4 报警规则

常见报警规则：

* 错误率阈值报警；
* 错误数阈值报警；
* 新增错误报警；

# 13. 总结

前端能力体系不是单点工具建设，而是一套围绕研发效率、交付质量、线上稳定性和用户体验的综合体系。

推荐建设顺序：

```txt
工程化打底
  ↓
质量保障卡口
  ↓
稳定性闭环
  ↓
性能体验治理
  ↓
架构能力沉淀
  ↓
AI 辅助研发提效
```

最终目标是：

```txt
开发更规范
协作更顺畅
构建更稳定
上线更可控
问题更快发现
故障更快恢复
体验持续优化
复杂项目长期可维护
```
