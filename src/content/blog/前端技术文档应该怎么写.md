一个优秀的前端技术文档，不是“写得很长”，而是能让 **开发、测试、后端、产品、AI/Codex** 都快速理解：

> 这个功能为什么做、怎么做、改哪里、风险在哪、怎么验证、以后怎么维护。

前端技术文档应该分成 **“能落地的工程文档”**，而不是纯说明书。

---

# 一、优秀前端技术文档的核心标准

## 1. 目标明确

文档开头必须回答：

```md
这个文档解决什么问题？
这个需求最终要交付什么？
不做什么？
```

例如：

```md
## 目标

实现 AI 对话页中的“需求澄清卡片”，支持：
- 单选题
- 多选题
- 上一题 / 下一题
- 确认提交
- 最终拼接为用户消息发送

本次不处理：
- 后端接口重构
- 历史消息兼容
- 语音输入逻辑
```

**优秀文档一定要写“不做什么”。**
否则 AI 或开发很容易扩大范围。

---

## 2. 背景清楚

不要只写“实现某功能”，要说明上下文。

例如：

```md
## 背景

当前 AI 对话页已经支持普通文本消息和语音消息。
现在需要在对话流中增加“需求澄清”流程。

当用户输入需求不完整时，后端会返回一组澄清问题。
前端需要以卡片形式展示问题，并支持用户逐题选择，最终把选择结果拼接成一条完整消息。
```

这能帮助开发理解为什么要这样设计，而不是机械写页面。

---

## 3. 页面流程要画清楚

前端文档一定要有 **页面流程 / 用户路径**。

例如：

```txt
用户进入聊天页
  ↓
输入需求
  ↓
点击发送
  ↓
后端判断需求不完整
  ↓
返回澄清问题
  ↓
前端展示 ClarifyCard
  ↓
用户逐题选择
  ↓
点击最终确认
  ↓
前端拼接完整消息
  ↓
继续调用发送消息接口
```

如果是复杂功能，最好补充状态流转：

```txt
idle → inputting → sending → clarifying → confirming → completed
                         ↓
                       error
```

---

## 4. UI 结构要拆组件

不要只贴 UI 图。要把 UI 拆成组件结构。

例如：

```txt
pages/chat/index.vue
  ├── ChatMessageList.vue
  │     └── ChatBubble.vue
  │           ├── TextMessage.vue
  │           ├── AudioMessage.vue
  │           └── ClarifyCard.vue
  └── ChatInputBar.vue
        ├── TextInput.vue
        ├── VoiceButton.vue
        └── ToolPanel.vue
```

每个组件要说明职责：

| 组件              | 职责                | 是否复用 | 是否有业务逻辑 |
| --------------- | ----------------- | ---- | ------- |
| ChatMessageList | 渲染消息列表            | 是    | 否       |
| ChatBubble      | 控制消息气泡样式          | 是    | 否       |
| ClarifyCard     | 展示澄清问题和选项         | 是    | 少量交互    |
| ChatInputBar    | 输入、发送、语音入口        | 是    | 局部交互    |
| chat store      | 管理消息、loading、当前状态 | 否    | 是       |

---

## 5. 状态必须写全

前端最容易返工的地方，不是 UI，而是状态没写清楚。

优秀文档要写：

```md
## ClarifyCard 状态

| 状态 | 触发条件 | UI 表现 | 是否可操作 |
|---|---|---|---|
| default | 初始状态 | 普通卡片 | 可选择 |
| selected | 用户已选择 | 选项高亮 | 可切换 |
| disabled | 已确认 | 选项置灰 | 不可操作 |
| loading | 提交中 | 按钮 loading | 不可重复点击 |
| error | 提交失败 | 展示错误提示 | 可重试 |
| completed | 全部问题完成 | 展示确认结果 | 不可编辑 |
```

尤其是这些状态必须明确：

```txt
loading
empty
error
disabled
selected
active
success
readonly
expired
```

## 6. 数据结构要提前定义

前端技术文档必须包含核心类型。

例如：

```ts
export interface ClarifyQuestion {
  id: string
  title: string
  type: 'single' | 'multiple'
  options: ClarifyOption[]
  required: boolean
}

export interface ClarifyOption {
  label: string
  value: string
}

export interface ClarifyAnswer {
  questionId: string
  values: string[]
}
```

不要等写代码时再让 AI 猜字段。

如果接口还没定，就写成：

```ts
// TODO: 等后端确认字段名
export interface ClarifyQuestionDTO {
  question_id: string
  question_text: string
  question_type: 'single' | 'multiple'
  options: Array<{
    label: string
    value: string
  }>
}
```

---

## 7. 接口约定要清楚

前端文档里要写：

```md
## 接口

### 发送消息

POST /chat/send

请求参数：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| content | string | 是 | 用户输入内容 |
| sessionId | string | 是 | 会话 ID |
| messageType | string | 是 | 消息类型 |

响应：

| 字段 | 类型 | 说明 |
|---|---|---|
| messageId | string | 消息 ID |
| replyType | text / clarify | 返回类型 |
| clarifyQuestions | ClarifyQuestion[] | 澄清问题 |
```

还要写异常情况：

```md
异常处理：
- 网络失败：toast 提示，恢复输入框
- 接口超时：允许用户重试
- 返回空问题：降级为普通文本回复
- 字段缺失：前端兜底展示“请补充更多信息”
```

---

## 8. 分层设计要明确

对你这种 uniapp + Vue3 + TypeScript + Pinia 项目，我建议文档必须写清楚分层。

```txt
api      只负责 HTTP 请求
service  负责业务流程编排
store    负责全局/页面共享状态
page     负责页面组织
component 负责 UI 展示和局部交互
types    负责类型定义
utils    负责纯函数工具
```

例如：

```txt
api/chat.ts
  - sendMessage()
  - getHistoryMessages()

services/chatService.ts
  - sendUserMessage()
  - buildClarifySummary()
  - normalizeMessageDTO()

stores/chat.ts
  - messages
  - sending
  - currentSessionId
  - appendMessage()
  - setSending()

components/chat/ClarifyCard.vue
  - props: questions
  - emits: confirm
```

---

## 9. 边界和风险要写出来

优秀文档一定有“风险与边界”。

例如：

```md
## 边界情况

1. 后端返回空问题列表
2. 用户快速连续点击确认
3. 用户中途退出页面
4. 消息发送中网络断开
5. 历史消息中包含旧版本 clarify 数据
6. 多选题未选择但 required = true
7. 选项文案过长导致 UI 换行
8. 小程序键盘弹起遮挡输入框
```

这部分对减少返工非常有用。

---

## 10. 验收标准要可测试

不要写：

```txt
功能正常
样式正常
体验良好
```

这种没法验收。

应该写：

```md
## 验收标准

1. 输入框为空时，发送按钮不可点击
2. 输入内容后，发送按钮可点击
3. 点击发送后，按钮进入 loading 状态
4. 后端返回 clarify 类型时，消息列表中展示 ClarifyCard
5. 单选题只能选择一个选项
6. 多选题可以选择多个选项
7. 必填题未选择时，点击下一题提示用户选择
8. 点击最终确认后，选项不可再编辑
9. 最终结果能拼接为一条用户消息
10. 所有新增代码通过 ESLint 和 TypeScript 检查
```

# 二、我推荐的前端技术文档模板

你可以直接按这个结构写：

```md
# 功能名称

## 1. 背景

为什么要做这个功能？当前问题是什么？

## 2. 目标

本次要实现什么？

## 3. 不做什么

明确排除范围，避免需求扩大。

## 4. 用户流程

用文字或流程图描述用户路径。

## 5. 页面结构

页面由哪些区域组成？

## 6. 组件拆分

列出组件、职责、props、emits。

## 7. 状态设计

列出 loading、empty、error、disabled、selected 等状态。

## 8. 数据结构

定义前端核心类型。

## 9. 接口设计

说明请求参数、响应字段、异常处理。

## 10. 分层设计

说明 api / service / store / page / component 的职责。

## 11. 实现步骤

按阶段拆任务。

## 12. 边界情况

列出可能出问题的场景。

## 13. 验收标准

列出可测试的标准。

## 14. 自测清单

开发完成后如何验证？

## 15. 风险点

哪些地方依赖后端、UI、接口、历史数据？
```

# 三、适合给 Codex / Cursor / Claude Code 的文档，要额外注意什么

如果文档是给 AI 写代码用的，重点不是“优雅”，而是 **约束明确**。

一定要写：

```md
## 修改范围

只允许修改：
- pages/chat/index.vue
- components/chat/ClarifyCard.vue
- stores/chat.ts
- types/chat.ts

不允许修改：
- 请求封装基类
- 全局样式
- 路由配置
- 其他无关页面
```

还要写：

```md
## 执行方式

1. 先阅读相关文件
2. 先输出实现计划
3. 不要立即改代码
4. 确认后再分步骤实现
5. 每次只完成一个阶段
6. 修改后说明变更文件和自测方式
```

给 AI 的文档要避免模糊词：

| 不推荐    | 推荐                            |
| ------ | ----------------------------- |
| 样式好看一点 | 按设计图实现，间距 24rpx，圆角 16rpx      |
| 交互正常   | 点击确认后按钮 loading，接口完成前不可重复点击   |
| 兼容一下   | 兼容空数据、接口失败、字段缺失               |
| 代码整洁   | 不修改无关文件，不超过组件职责，复杂逻辑放 service |
| 优化性能   | 列表使用分页渲染，避免 watch 深层对象        |

# 四、文档写到什么程度才算够？

我的判断标准是：

```txt
一个新人拿到文档，可以知道怎么开发；
一个测试拿到文档，可以知道怎么测；
一个后端拿到文档，可以知道接口怎么配合；
一个 AI 拿到文档，可以少问问题、少乱改代码；
一个月后你自己回来，可以知道当时为什么这么设计。
```

满足这五点，就是好文档。

---

# 五、最实用的结论

优秀前端技术文档应该做到：

```txt
说清目标
画清流程
拆清组件
定义状态
约定数据
明确接口
限制范围
写出边界
给出验收
方便 AI 执行
```

写前端需求文档，固定用这个结构：

```txt
背景 → 目标 → 不做什么 → 流程 → 组件 → 状态 → 数据 → 接口 → 分层 → 步骤 → 边界 → 验收
```

尤其是给 Codex 用时，最重要的是三件事：

1. **限制它改哪些文件。**
2. **明确组件和状态。**
3. **让它先出计划，再写代码。**
