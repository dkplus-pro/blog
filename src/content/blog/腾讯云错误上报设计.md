# 腾讯云前端错误上报与告警设计实践

## 一、背景

前端项目上线后，最怕的问题不是“有错误”，而是：

* 错误发生了，但没人知道；
* 收到了告警，但不知道影响哪个业务；
* 告警很多，但无法判断优先级；
* 后端让提供 `requestId`，前端没有记录；
* 报错信息太散，无法形成稳定的告警规则。

因此，前端监控不能只是简单地把 `error.message` 上报上去，而是要设计一套**结构化的错误上报和告警体系**。

腾讯云前端监控支持额外参数，例如 `ext1`、`ext2`、`ext3` 等字段。我们可以利用这些字段，把错误等级、业务场景、错误类型、页面、版本、接口等信息标准化，从而实现更准确的告警、聚合和排查。

---

## 二、核心设计原则

### 1. 报错信息不直接作为告警规则

很多前端错误信息是不稳定的，例如：

```text
Cannot read properties of undefined
Network Error
接口请求失败
timeout of 10000ms exceeded
```

这些错误信息虽然对排查有帮助，但不适合作为告警规则。

原因是：

* 文案容易变化；
* 同类错误可能有很多种不同描述；
* 无法稳定区分业务影响；
* 告警容易被打散；
* 不方便做统计和收敛。

所以，错误信息应该作为**问题详情**，而不是作为**告警维度**。

---

### 2. 告警规则应该基于结构化字段

真正适合作为告警规则的是这些字段：

```ts
severity: 'P0' | 'P1' | 'P2' | 'P3';
bizScene: string;
errorType: string;
module: string;
env: string;
version: string;
route: string;
apiName?: string;
owner?: string;
channel?: string;
```

也就是说：

> `msg / stack / requestId` 负责排查问题，`ext1 ~ ext10` 负责告警聚合和规则匹配。

---

## 三、推荐 ext 字段设计

腾讯云支持多个额外参数，例如 `ext1`、`ext2` 等。建议不要随意使用，而是提前定义统一规范。

推荐映射如下：

| 字段      | 含义       | 示例                        | 用途       |
| ------- | -------- | ------------------------- | -------- |
| `ext1`  | 错误等级     | `P0` / `P1` / `P2` / `P3` | 告警优先级    |
| `ext2`  | 业务场景     | `chat_send_message`       | 判断影响哪个业务 |
| `ext3`  | 错误类型     | `api_error` / `js_error`  | 区分错误来源   |
| `ext4`  | 业务模块     | `chat` / `share` / `clip` | 聚合模块问题   |
| `ext5`  | 环境       | `prod` / `gray` / `test`  | 区分线上和测试  |
| `ext6`  | 版本号      | `0.10.17`                 | 判断是否发版引入 |
| `ext7`  | 页面路径     | `/pages/chat/index`       | 定位具体页面   |
| `ext8`  | 接口名称     | `sendMessage`             | 定位具体接口   |
| `ext9`  | 负责人 / 团队 | `frontend-chat`           | 告警路由     |
| `ext10` | 渠道 / 灰度  | `wx_mp` / `gray_10`       | 判断影响范围   |

## 四、代码里需要上报哪些维度

### 1. 错误等级 severity

错误等级是告警系统最核心的字段。

建议分为：

| 等级   | 含义                 | 示例                            |
| ---- | ------------------ | ----------------------------- |
| `P0` | 核心链路不可用，严重影响用户     | 登录失败、支付失败、聊天发送失败、验证码失败、核心页面白屏 |
| `P1` | 重要功能异常，影响部分用户或部分流程 | 消息重连失败、本地音乐上传失败、关键数据加载失败      |
| `P2` | 普通功能异常，不阻塞核心流程     | UI 偶发错位、非核心资源加载失败             |
| `P3` | 低优先级问题             | warning、埋点异常、非阻塞型异常           |

示例：

```ts
severity: 'P0'
```

---

### 2. 业务场景 bizScene

业务场景一定要用枚举，不要让开发随手写中文描述。

推荐：

```ts
type BizScene =
  | 'sms_code_login'
  | 'chat_send_message'
  | 'chat_reconnect'
  | 'companion_yesterday_summary'
  | 'music_upload_local'
  | 'share_identity_verify'
  | 'payment_member_buy';
```

这样告警规则可以直接配置：

```text
ext1 = P0
ext2 = sms_code_login
```

或者：

```text
ext1 = P0
ext2 = chat_send_message
```

这样收到告警时，就能立刻知道：

* 是不是核心功能；
* 影响哪个业务；
* 应该找谁处理；
* 是否需要立刻回滚或止血。

---

### 3. 错误类型 errorType

错误类型用于区分错误来源。

推荐：

```ts
type ErrorType =
  | 'js_error'
  | 'promise_error'
  | 'api_error'
  | 'retcode_error'
  | 'resource_error'
  | 'white_screen'
  | 'timeout'
  | 'websocket_error'
  | 'business_error';
```

常见分类说明：

| 错误类型              | 说明                |
| ----------------- | ----------------- |
| `js_error`        | JS 运行时报错          |
| `promise_error`   | 未捕获的 Promise 异常   |
| `api_error`       | 接口请求失败            |
| `retcode_error`   | 接口正常返回，但业务状态码异常   |
| `resource_error`  | 图片、JS、CSS 等资源加载失败 |
| `white_screen`    | 页面白屏              |
| `timeout`         | 请求超时              |
| `websocket_error` | WebSocket 连接异常    |
| `business_error`  | 业务逻辑异常            |

---

### 4. requestId / traceId

`requestId` 非常重要，但不建议作为告警规则字段。

原因是：

* 每次请求的 `requestId` 都不同；
* 如果放进告警维度，会导致告警被打散；
* 它更适合用于排查链路，而不是用于聚合告警。

推荐上报：

```ts
requestId: 'xxx';
traceId: 'xxx';
spanId: 'xxx';
```

典型排查链路：

```text
前端收到告警
    ↓
查看错误详情
    ↓
拿到 requestId / traceId
    ↓
查询后端日志
    ↓
定位接口、参数、服务、数据库或第三方依赖问题
```

---

### 5. 接口维度

接口错误是前端告警中非常常见的一类。

建议上报：

```ts
apiName: 'sendMessage';
url: '/api/chat/send';
method: 'POST';
status: 500;
retCode: 10023;
duration: 3120;
requestId: 'xxx';
```

这里有一个重要细节：

接口地址最好做归一化，不要带动态参数。

不要这样上报：

```text
/api/order/123456/detail
/api/order/789101/detail
```

应该统一成：

```text
/api/order/:id/detail
```

否则同一个接口会被拆成很多不同的 URL，影响聚合和告警判断。

---

### 6. 页面和路由

前端错误一定要带页面信息。

推荐上报：

```ts
route: '/pages/chat/index';
pageName: 'chat';
fromPage: '/pages/home/index';
```

这样可以快速回答几个问题：

* 哪个页面出错最多？
* 是不是某个入口跳转后才出问题？
* 是不是只有某个页面版本有问题？
* 是不是某个页面白屏或接口失败率异常？

---

### 7. 版本信息

版本信息必须上报。

推荐：

```ts
version: '0.10.17';
buildId: '20260618-001';
commitId: 'a1b2c3d';
releaseChannel: 'gray';
```

线上出问题时，第一时间要判断：

* 是不是刚发版导致的？
* 是不是只影响某个版本？
* 是不是灰度版本有问题？
* 是否需要回滚？
* 是否只影响某个小程序 app？

---

### 8. 用户和设备信息

推荐上报：

```ts
userId: hashUserId;
deviceId: deviceId;
sessionId: sessionId;
platform: 'wx_mp';
os: 'iOS';
brand: 'iPhone';
network: 'wifi';
```

注意：

不要直接上报手机号、身份证、真实姓名、openid 等敏感信息。

如果需要用户标识，建议使用脱敏后的 `userId` 或内部匿名 ID。

## 五、统一上报方法设计

建议不要在业务代码里直接调用腾讯云 SDK，而是封装一层统一方法。

示例：

```ts
type ErrorLevel = 'P0' | 'P1' | 'P2' | 'P3';

type ErrorType =
  | 'js_error'
  | 'promise_error'
  | 'api_error'
  | 'retcode_error'
  | 'resource_error'
  | 'white_screen'
  | 'timeout'
  | 'websocket_error'
  | 'business_error';

interface ReportErrorParams {
  severity: ErrorLevel;
  bizScene: string;
  errorType: ErrorType;
  module: string;
  route: string;

  msg: string;
  stack?: string;

  apiName?: string;
  url?: string;
  method?: string;
  status?: number;
  retCode?: string | number;
  duration?: number;

  requestId?: string;
  traceId?: string;

  extra?: Record<string, any>;
}

export function reportFrontendError(params: ReportErrorParams) {
  const {
    severity,
    bizScene,
    errorType,
    module,
    route,
    msg,
    stack,
    apiName,
    url,
    method,
    status,
    retCode,
    duration,
    requestId,
    traceId,
    extra,
  } = params;

  aegis.error({
    msg: JSON.stringify({
      msg,
      stack,
      url,
      method,
      status,
      retCode,
      duration,
      requestId,
      traceId,
      extra,
    }),

    // 告警规则主要基于这些字段
    ext1: severity,
    ext2: bizScene,
    ext3: errorType,
    ext4: module,
    ext5: getEnv(),
    ext6: getAppVersion(),
    ext7: route,
    ext8: apiName || 'none',
    ext9: getOwnerByModule(module),
    ext10: getReleaseChannel(),

    // 链路定位
    trace: traceId || requestId || '',
  });
}
```

## 七、告警规则建议

### 1. P0 告警

P0 应该非常敏感。

建议规则：

```text
ext1 = P0
ext5 = prod
5 分钟内错误数 >= 1
```

适用场景：

* 登录失败；
* 验证码失败；
* 支付失败；
* 聊天发送失败；
* 核心页面白屏；
* 核心接口 5xx；
* 核心业务数据无法加载。

P0 告警建议直接通知：

* 企业微信群；
* 值班人；
* 负责人；
* 必要时短信或电话。

---

### 2. P1 告警

P1 可以稍微收敛一点，避免过度打扰。

建议规则：

```text
ext1 = P1
ext5 = prod
5 分钟内错误数 >= 5
```

或者：

```text
ext1 = P1
ext5 = prod
影响用户数 >= 3
```

适用场景：

* 消息重连失败；
* 上传失败；
* 重要页面局部接口异常；
* 非核心但高频使用的功能异常。

---

### 3. 业务场景告警

#### 验证码获取失败

```text
ext2 = sms_code_login
ext3 = api_error
ext5 = prod
5 分钟内错误数 >= 3
```

#### 聊天消息发送失败

```text
ext2 = chat_send_message
ext3 = api_error
ext5 = prod
5 分钟内错误数 >= 3
```

#### 页面白屏

```text
ext3 = white_screen
ext5 = prod
错误数 >= 1
```

#### 本地音乐上传失败

```text
ext2 = music_upload_local
ext3 = api_error
ext5 = prod
10 分钟内错误数 >= 5
```

## 九、不要做的事情

### 1. 不要把错误文案当成唯一告警依据

错误文案可以辅助排查，但不要作为主要规则。

错误示例：

```text
msg contains "Network Error"
```

更好的方式：

```text
ext1 = P0
ext2 = chat_send_message
ext3 = api_error
```

---

### 2. 不要把 requestId 放进告警维度

`requestId` 是排查字段，不是聚合字段。

错误做法：

```text
ext8 = requestId
```

正确做法：

```text
ext8 = apiName
trace = requestId
msg.detail.requestId = requestId
```

---

### 3. 不要让业务场景自由发挥

错误示例：

```ts
bizScene: '验证码失败';
bizScene: '获取验证码失败';
bizScene: '登录验证码报错';
bizScene: 'sms error';
```

正确做法：

```ts
bizScene: 'sms_code_login';
```

业务场景必须枚举化，否则告警规则会越来越乱。

---

### 4. 不要所有错误都按 P0 报警

如果所有错误都是 P0，最后所有告警都会被忽略。

P0 只给真正阻断核心流程的问题。

## 十、推荐落地步骤

### 第一步：定义字段规范

先确定 `ext1 ~ ext10` 的含义。

```ts
const EXT_MAP = {
  ext1: 'severity',
  ext2: 'bizScene',
  ext3: 'errorType',
  ext4: 'module',
  ext5: 'env',
  ext6: 'version',
  ext7: 'route',
  ext8: 'apiName',
  ext9: 'owner',
  ext10: 'channel',
};
```

---

### 第二步：定义业务场景枚举

```ts
export const BIZ_SCENE = {
  SMS_CODE_LOGIN: 'sms_code_login',
  CHAT_SEND_MESSAGE: 'chat_send_message',
  CHAT_RECONNECT: 'chat_reconnect',
  COMPANION_YESTERDAY_SUMMARY: 'companion_yesterday_summary',
  MUSIC_UPLOAD_LOCAL: 'music_upload_local',
  SHARE_IDENTITY_VERIFY: 'share_identity_verify',
  PAYMENT_MEMBER_BUY: 'payment_member_buy',
} as const;
```

---

### 第三步：封装统一上报方法

所有业务代码都走：

```ts
reportFrontendError();
```

不要散落在业务代码里直接调用腾讯云 SDK。

---

### 第四步：接入请求封装

在请求层统一处理：

* 请求失败；
* HTTP 状态码异常；
* 业务 retCode 异常；
* 请求超时；
* requestId / traceId 提取；
* 接口耗时；
* 接口名称归一化。

---

### 第五步：配置腾讯云告警规则

优先配置：

1. P0 线上错误；
2. 核心业务场景错误；
3. 核心接口失败；
4. 页面白屏；
5. 新版本错误率上升；
6. 灰度版本异常。

---

### 第六步：建立告警处理流程

一条告警发出来后，应该能回答：

* 什么业务出问题？
* 严重等级是多少？
* 哪个版本？
* 哪个页面？
* 哪个接口？
* 影响多少用户？
* 负责人是谁？
* requestId 是什么？
* 是否需要回滚？

---

## 十一、最终总结

前端监控的关键，不是“把错误报上去”，而是“把错误报得有结构”。

推荐原则：

```text
ext 字段负责告警和聚合
msg 字段负责错误详情
requestId / traceId 负责链路排查
severity 负责判断优先级
bizScene 负责判断业务影响
version 负责判断是否发版引入
owner 负责判断谁来处理
```

最终目标是：

> 告警一出现，就能知道影响范围、业务场景、责任人和排查入口，而不是只看到一条模糊的 `Network Error`。
