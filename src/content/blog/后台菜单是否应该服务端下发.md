一般建议：**管理后台左侧菜单由服务端下发**，前端只负责渲染和路由匹配。

尤其是你做的是后台管理系统，有用户、角色、权限这些概念时，更推荐服务端控制菜单。

---

## 结论先说

| 方案                 | 适合场景                |
| ------------------ | ------------------- |
| 前端本地写死菜单           | 小项目、个人项目、没有权限系统     |
| 服务端下发菜单            | 企业后台、多角色、多权限、菜单动态变化 |
| 前端静态路由 + 服务端下发权限码  | 中大型项目常见方案           |
| 服务端下发完整菜单 + 前端动态路由 | 权限强控制、菜单高度动态的后台     |

最推荐的是：

```text
服务端维护菜单和权限
前端维护页面组件和基础路由映射
登录后服务端返回当前用户可见菜单
前端根据菜单渲染左侧栏
```

---

## 为什么推荐服务端下发？

### 1. 权限更安全

如果菜单写死在前端，只是前端控制“显示不显示”。

但前端代码用户是可以看到的，不能作为真正的权限控制。

正确做法应该是：

```text
前端：控制菜单是否显示
后端：控制接口是否能访问
```

也就是说：

```text
菜单权限：服务端下发
接口权限：服务端校验
按钮权限：服务端下发权限码，前端控制显示
```

---

### 2. 多角色更方便

比如：

```text
超级管理员：用户管理、角色管理、订单管理、系统设置
运营人员：订单管理、商品管理
财务人员：账单管理、提现管理
普通员工：只能看数据看板
```

如果菜单写在前端，你会写很多判断：

```js
if (role === 'admin') { ... }
if (role === 'finance') { ... }
if (role === 'operator') { ... }
```

后期会越来越乱。

服务端下发就简单：

```text
用户登录
↓
后端根据用户角色查菜单
↓
只返回这个用户能看的菜单
↓
前端直接渲染
```

## 推荐架构

### 后端负责这些

```text
菜单表
角色表
用户表
权限表
角色-菜单关系
角色-权限关系
```

菜单表可以类似这样：

```text
id
parent_id
name
path
component
icon
sort
type
permission
visible
```

其中：

```text
name：菜单名称
path：前端路由路径
component：前端组件标识
icon：图标
sort：排序
type：目录 / 菜单 / 按钮
permission：权限标识，比如 user:list
visible：是否显示
```

---

### 前端负责这些

```text
路由组件映射
菜单渲染
动态添加路由
按钮权限判断
无权限页面处理
```

前端不要完全信任服务端传来的 `component` 字符串直接随便加载，而是做一个映射表。

例如：

```js
const componentMap = {
  'system/user/index': () => import('@/views/system/user/index.vue'),
  'system/role/index': () => import('@/views/system/role/index.vue'),
  'order/list/index': () => import('@/views/order/list/index.vue')
}
```

服务端返回：

```json
{
  "name": "用户管理",
  "path": "/system/user",
  "component": "system/user/index",
  "permission": "user:list",
  "icon": "User"
}
```

前端根据 `component` 找到真实 Vue 组件。

## 常见方案：登录后获取菜单

登录流程：

```text
用户登录
↓
后端返回 token
↓
前端用 token 请求 /user/info 或 /menus
↓
后端返回用户信息、角色、权限、菜单树
↓
前端保存菜单
↓
前端动态生成路由
↓
渲染左侧菜单
```

后端返回示例：

```json
{
  "user": {
    "id": 1,
    "username": "admin"
  },
  "roles": ["admin"],
  "permissions": ["user:list", "user:add", "user:delete"],
  "menus": [
    {
      "name": "系统管理",
      "path": "/system",
      "icon": "Setting",
      "children": [
        {
          "name": "用户管理",
          "path": "/system/user",
          "component": "system/user/index",
          "permission": "user:list"
        },
        {
          "name": "角色管理",
          "path": "/system/role",
          "component": "system/role/index",
          "permission": "role:list"
        }
      ]
    }
  ]
}
```

## 什么时候可以写在前端本地？

下面这些情况可以前端写死：

```text
项目很小
只有一个管理员角色
没有复杂权限
菜单基本不会变
只是内部工具
后端还没做权限系统
```

比如个人后台、毕业设计、小 Demo，前端本地写菜单没问题。

但如果你后面要做：

```text
角色管理
权限管理
菜单管理
按钮权限
不同用户看到不同菜单
```

那就应该服务端下发。

---

## 最佳实践

我建议你采用这个组合：

```text
菜单结构：服务端下发
页面组件：前端本地映射
接口权限：后端强校验
按钮权限：服务端返回权限码，前端判断显示
```

这样比较稳。

不要做成：

```text
前端写死所有菜单
只靠前端判断角色
```

因为用户可以绕过页面直接请求接口。

也不要完全做成：

```text
后端下发任意 component 路径，前端直接 import
```

这样不安全，也容易打包失败。

---

## 一句话总结

**管理后台左侧菜单，正式项目建议服务端下发；前端只做渲染和组件映射。**

最合理的分工是：

```text
服务端决定：当前用户能看哪些菜单、拥有哪些权限
前端决定：这些菜单怎么显示、对应哪个 Vue 页面
后端接口再做真正的权限校验
```
